## L-001 — MPP SessionManager.log Rotation Is Fast (~1 hour) Under POM Campaign Load; CCXML Per-Slot Logs Last Longer

- **Symptom**: MPP tarball captured >1 hour after a POM incident does not contain the failing call's SIP-level evidence in `SessionManager.log` rotation — log retention window has cycled past the incident timestamp.
- **Evidence**: For SR 1-23647477802 BC2 (call at 10:14:15 JST, MPP `PVCLIPOMPP2A21H` tarball captured 11:40:15 JST = ~1.5 hour later): all 8 `SessionManager.log` rotation files (`.log`, `.log.1`...`.log.7`, each ~21 MB) contained 0 BC2 UCID/SID/phone hits. But `CCXML-SessionSlot-011.log.2` and `CCXML-SessionSlot-020.log.1` for the same call retained 2276 SID hits + 87 phone hits — driver session + nail-up session CXI logs still intact. BC1 (`PVCLIPOMPP2A11H` tarball captured ~25 min after call) had full SessionManager.log SIP evidence at the time of capture.
- **Root cause**: `SessionManager.log` rotation is size-based (~21 MB per file × 8 files retained = ~168 MB total). Under typical POM campaign load with multiple concurrent calls, total SIP volume reaches that size within ~1 hour. CCXML per-slot logs rotate per-session (one logical file per call session), so per-call data is preserved for the lifetime of that slot file regardless of clock time.
- **Fix / workaround**: For POM incidents, request `getmpplogs.sh` tarball within ~1 hour of the incident. If too late, fall back to CCXML per-slot logs which retain session-level events including `hints.sip.replaces` and `AGTCallNotifyDriver`. The slot ID is decoded from Platform Session ID (Campaign Report column 33) trailing `-NN`: e.g. `PVCLIPOMPP2A21H-2026152011401-11` → `CCXML-SessionSlot-011.log*`.

## L-002 — `hints.sip.replaces` in CXI Log Is Equivalent Evidence to `SND ^INVITE` with Replaces in SessionManager.log

- **Symptom**: When MPP `SessionManager.log` SIP-level evidence has rotated out (per L-001), need to prove the Replaces INVITE was actually sent without the actual wire-level SIP message.
- **Evidence**: For BC2 (SessionManager.log rotated past BC2 timestamp), found in `CCXML-SessionSlot-020.log.1` at 2026-06-01 10:14:28.481: `hints.sip.replaces: "8251c4d6c45bf11c3c90011ac2c3e;to-tag=342c779e5d5741f1a57f0505695906f;from-tag=6e51c4d6c45bf11c2c90011ac2c3e"`. Pattern: AgtMgr sends AGTCallCustomer → DriverApp → MPP nail-up CXI; CXI Interpreter generates `hints.sip.replaces` event; SessionManager IMMEDIATELY generates outbound SIP INVITE with that Replaces header (~18-20ms later, confirmed by whistle timestamp). For BC1 where both logs were preserved, the timing relationship was directly observable: CXI `hints.sip.replaces` event triggers SessionManager `SND ^INVITE` consistently.
- **Root cause**: `hints.sip.replaces` is the internal CXI Interpreter event that causes `command.createcall` with a Replaces header parameter, which in turn causes SessionManager to emit the SIP INVITE. POM CXI does not emit this event without immediately invoking createcall — there is no path in the CCXML state machine that aborts between the two.
- **Fix / workaround**: When SessionManager.log doesn't preserve the failing call, accept CXI log `hints.sip.replaces` as equivalent evidence that the Replaces INVITE was generated by MPP. This proves the MPP side of the bridging — but does NOT by itself diagnose any downstream behavior. Combine with direct RTP packet-count evidence at the customer-edge SBC boundary (per sip-voice-quality L-002) before drawing conclusions about media-plane outcomes.

## L-003 — SM Syslog Stream Is an Independent SIP Transaction Record — Useful for Pcap Cross-Validation

- **Symptom**: Need an independent SIP transaction record to corroborate or refute observations made from customer-side pcap (which may have coverage gaps, dropped packets, or capture-point limitations).
- **Evidence**: For SR `1-23647477802`, SM `PVCLIPBASM0031H` at 172.17.61.10 forwards every SIP transaction it processes to UDP `172.17.65.96:514` (and `.97:514` for HA). Each syslog packet contains the full SIP message in payload (sometimes spanning 2-3 UDP packets, grouped by `(src_port, dst_port)` tuple). The SM syslog stream is visible inside the customer pcap because the syslog traffic transits the same capture point. Reconstructing SIP messages from this stream provides a second, independent source for what SM actually processed.
- **Root cause / mechanism**: SM logs every SIP message at SM ingress before any routing decision, providing a complete record of what SM saw. Useful as cross-validation when pcap evidence alone is inconclusive. Caveat: absence of a SIP message in SM syslog confirms only that SM did not process it — it does NOT by itself diagnose any specific component as the cause of that absence (e.g. CM may not have sent, SM may have failed to log, upstream signaling state may have not required the message at all).
- **Fix / workaround**: In customer pcaps, look for high-volume UDP from SM IP to remote `:514` / non-standard log ports — that's the syslog stream. Reconstruct full SIP messages by concatenating packet payloads grouped by `(src_port, dst_port)`. Use as cross-validation for pcap observations; do NOT use as sole evidence for any single-component root-cause conclusion.
